# LLM Worker Tool Catalog (詳細仕様)

本ドキュメントは、ADKベースの LLM worker が使用する各ツールの技術的な動作仕様である。

---

## 1. コア・処理ツール

### `extract_text`
- **目的**: 多様なファイル形式から構造を維持したままテキストを取り出す。
- **巨大データ対応**: 100MB を超える場合は、このツール自体が「ファイルを分割して保存」し、分割されたメタデータのリストを返す。

### `semantic_chunking`
- **目的**: LLM が一度に扱えるサイズに、文脈を壊さずに分割する。
- **Schema**: RawText -> Chunks[], Outline[]

### `goal_driven_synthesis`
- **目的**: ドキュメントの論理構造を解析し、ナレッジツリーを再構築する。
- **自律性**: LLM がドキュメントの性質に合わせて最適なアイテム分類と階層を決定する。

### `generate_html_summary`
- **目的**: アイテム解説文を読みやすい HTML で生成する。子アイテムへのリンクを自動埋込する。

### `extract_table_data`
- **目的**: 複雑な表構造を壊さずに JSON/Markdown 形式でパースする。
- **用途**: 財務諸表や比較表などの正確な数値解析。

---

## 2. 管理・インフラツール

### `analyze_dependencies`
- **目的**: 情報の論理的な依存関係（前提知識）を特定し、最適な処理順序を決定する。

### `manage_job_checklist`
- **目的**: ジョブの進捗と `depends_on` 関係を管理する。
- **巨大データ対応**: 全体の処理工程をサブタスク化して追跡・再開を可能にする。

### `manage_glossary`
- **目的**: 文書特有の用語定義を記憶し、解析全体での解釈の一貫性を保つ。

### `repair_encoding`
- **目的**: テキストの文字化け（Mojibake）や不適切なエンコーディングを修正する。
- **用途**: 解析精度を落とす「ノイズ」の除去。

### `persist_knowledge_tree`
- **目的**: 生成されたアイテム群を DB に永続化する。

---

## 3. 品質・高度な処理ツール

### `quality_critique` (自己検閲)
- **目的**: 生成された結果を自ら監査し、幻覚（Hallucination）や論理矛盾を指摘する。
- **重要性**: エージェントが「自分のミス」を修正して、最終的な精度を商用レベルまで引き上げる。

### `deduplicate_and_merge`
- **目的**: 分割処理（Map-Reduce）の結果発生した重複した概念を一つに統合する。
- **用途**: きれいに洗練されたツリー構造の維持。

### `semantic_search` (自律型RAG)
- **目的**: 巨大なデータ、あるいはワークスペース全体のドキュメントの中から特定の情報をピンポイントで検索・抽出する。
- **用途**: 
    - 現在処理中のファイルだけでなく、同じワークスペースにある「過去の知識」を横断して検索。
    - 1GB級のドキュメントにおいて、全文を読み直さずに特定の主張や数値の根拠を再確認する。
- **技術的背景**: Postgres (PGVector) を用いたベクトル検索により、ワークスペース単位のフィルタリングと高速検索を両立。

### `summarize_batch`
- **目的**: 大量のチャンクを並列で要約・圧縮する。
