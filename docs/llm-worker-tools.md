# LLM Worker Tool Catalog (詳細仕様)

本ドキュメントは、ADKベースの LLM worker が使用する各ツールの技術的な動作仕様である。

---

## 1. コア・処理ツール

### `extract_text`
- **目的**: 多様なファイル形式から構造を維持したままテキストを取り出す。
- **技術詳細**: 
    - PDF の場合は OCR または PDF パースライブラリを使用。
    - 巨大ファイルの場合、GCS からストリーム読み込みを行い、一時ファイルとして保存。
- **巨大データ対応**: 100MB を超える場合は、このツール自体が「ファイルを分割して保存」し、分割されたメタデータのリストを返す。
- **Schema**:
    - 入力: `{ "file_uri": "gs://...", "mime_type": "application/pdf" }`
    - 出力: `{ "raw_text_parts": ["gs://tmp/part1.txt", "gs://tmp/part2.txt"], "total_chars": 1200000 }`

### `semantic_chunking`
- **目的**: LLM が一度に扱えるサイズ（約 8k-16k tokens）に、文脈を壊さずに分割する。
- **LLM プロンプト戦略**: 「トピックが完全に切り替わる箇所を見つけ、そこに `---` の境界線を引け。各塊には内容を端的に表す見出しを付けろ」
- **Schema**:
    - 入力: `{ "text_uri": "gs://tmp/part1.txt" }`
    - 出力: `{ "chunks": [{ "index": 0, "heading": "第1章: 背景", "text": "..." }], "outline": ["背景", "課題"] }`

### `generate_brief`
- **目的**: 全体像を把握し、詳細な構造化（Synthesis）時の「ブレ」を防ぐためのガードレール。
- **動作**: チャンクごとの要約をさらに要約し、ドキュメントの「核」を抽出する。
- **Schema**:
    - 入力: `{ "outline": [...], "chunk_summaries": [...] }`
    - 出力: `{ "topic": "2024年度決算報告", "claim": "増収減益だが成長投資は継続", "key_entities": ["設備投資", "為替影響"] }`

### `goal_driven_synthesis`
- **目的**: ドキュメントの論理構造を解析し、情報を最も深く理解しやすく整理された「ナレッジツリー」として再構築する。
- **動作**: 
    - `Document Brief` で定義されたトピックに基づき、ドキュメントの性質（技術仕様、論文、法的文書など）に合わせて LLM が自ら最適なアイテムの分類と階層を決定する。
    - 単なる章立ての模倣ではなく、情報の包含関係や論理的な依存関係を重視した階層構造を合成する。
- **LLM プロンプト戦略**: 「ドキュメントの本質を効率的に伝え、Paper-in-Paper UI での探索体験を最大化するツリー構造を自律的に設計せよ。階層の深さやアイテムの粒度は情報の複雑度に応じて最適化し、すべての項目は必ず元のチャンクに根拠（source_chunk_ids）を持たせろ」

### `generate_html_summary`
- **目的**: ユーザーがアイテムをクリックした際に表示される解説文を、読みやすい HTML で生成する。
- **動作**: 
    - アイテムの説明だけでなく、子アイテムへのリンク（`<a data-paper-id="...">`）を文中に埋め込む。
    - 関連する図表や数値を `<table>` で整形。

## 2. 管理・インフラツール

### `analyze_dependencies`
- **目的**: ドキュメントの構成（Outline）を分析し、情報の論理的な依存関係を特定する。
- **役割**: 「第5章を理解するには第2章の定義が必須」といった関係を抽出することで、巨大データの最適な処理順序を決定する。
- **Schema**:
    - 入力: `{ "outline": ["Introduction", "Definitions", "Analysis"] }`
    - 出力: `{ "dependencies": [{ "task_id": "T3", "depends_on": "T2", "reason": "Requires terminology" }] }`

### `manage_job_checklist` (巨大データ対応の要)
- **目的**: エージェントの「外部メモリ」として、ジョブの進捗と依存関係を管理する。
- **特徴**: 
    - タスク間の `depends_on` をサポート。
    - `get_next_pending` により、前提条件が満たされたタスクのみを自動で抽出。
- **ユースケース**: クラッシュ時や巨大な並列ジョブにおいて、何をどこまでやったかを正確に把握する。
- **Schema**:
    - 操作: `add | update | list | get_next_pending`
    - データ: `{ "task_id": "T1", "description": "...", "status": "pending", "depends_on": ["T0"] }`

### `persist_knowledge_tree`
- **目的**: LLM が生成した一時的なアイテム群を、DB（Postgres）の `tree_items` 表に一括挿入する。
- **動作**: 
    - トランザクション管理。
    - ローカルID（LLMが振った名前）と実際の UUID（DBが振るID）の紐付け解決。
    - 完了時にジョブステータスを `completed` に変更。

## 3. 高度なツール (将来)

### `vector_search` (RAG用)
- **目的**: 1GB の中から「特定の情報」を高速に見つける。
- **動作**: 全テキストをベクトル化してインデックスを作成。エージェントが「この計算の根拠を探して」と投げると、数行のテキストを返す。

### `summarize_batch`
- **目的**: 100個のチャンクを並列でLLMに投げ、それぞれの要約を一気に生成する（Map処理）。
