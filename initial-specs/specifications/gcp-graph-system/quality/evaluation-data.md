# 13. Evaluation Data

## Overview

LLM を中核とするパイプラインでは、コードの正しさとは別に「出力品質」を継続的に測る仕組みが必要になる。本章では、各ステージの出力品質を評価するためのデータ設計・作成方針・運用サイクルを定義する。

---

## 評価対象と指標

### ノード抽出（Pass 1 + Pass 2）

| 評価項目 | 指標 |
| --- | --- |
| ノードの網羅性 | 正解ノードのうち何割が抽出されたか（Recall） |
| ノードの正確性 | 抽出ノードのうち何割が正解か（Precision） |
| level 割り当て | 正解 level との一致率 |
| category 割り当て | `concept` / `entity` / `claim` / `evidence` / `counter` の正解率 |
| claim-evidence 対応 | 正解ノード群のうち何割が正しくまとまりとして抽出されたか |

### HTML サマリ

| 評価項目 | 指標 |
| --- | --- |
| 出典整合性 | サマリの内容が出典チャンクと矛盾しないか |
| 情報カバレッジ | ノードの主要情報を網羅しているか |
| 構造品質 | 表・リストが適切に使われているか |

### トピックマッピング

| 評価項目 | 指標 |
| --- | --- |
| マッピング精度 | 正解トピックへのマッピング率 |
| 未マッピング率 | マッピングされるべきノードが漏れた割合 |

### ノード統合

| 評価項目 | 指標 |
| --- | --- |
| 統合漏れ | 統合すべきペアが統合されなかった割合 |
| 誤統合 | 統合すべきでないペアが統合された割合 |

---

## データの種類

### Gold Document（抽出評価用）

人手でアノテートした文書と、そこから期待されるノードのセット。

```
gold/
└─ doc_001/
   ├─ input.pdf
   ├─ expected_nodes.json
   └─ notes.md
```

**作成方針**
- 初期は 12 文書を人手でアノテートする
- 文書種別（企画書・仕様書・議事録・CSV）ごとに 3 文書ずつ用意する
- アノテーターが複数いる場合は一致率（Cohen's kappa 等）を記録する

### Synonym Pairs（ノード統合評価用）

統合すべき表記揺れペアと、統合すべきでないペアの対のリスト。

```json
[
  {"a": "販売戦略", "b": "セールス戦略", "should_merge": true},
  {"a": "販売戦略", "b": "マーケティング戦略", "should_merge": false}
]
```

### Topic Mapping Ground Truth（トピックマッピング評価用）

文書とトピックの正解対応。

```json
[
  {"document_id": "doc_001", "expected_topics": ["販売戦略", "新規開拓"]}
]
```

### HTML Summary Ratings（サマリ評価用）

人手または LLM-as-judge による評価スコア。

```json
{
  "node_id": "n_001",
  "source_accuracy": 4,
  "coverage": 3,
  "structure_quality": 5,
  "evaluator": "llm",
  "evaluated_at": "2026-03-28"
}
```

## データ作成フロー

### 初期（手作業）

1. 代表的な文書を 12 件選定する
2. パイプラインを一度実行して出力を得る
3. `dev` ロールの担当者 2 名で出力を確認・修正して正解データとする
4. 修正内容をアノテーションとして保存する

### 初期担当と工数

- 初期 gold 作成は `dev` ロール 2 名が担当する
- 1 文書あたり 30〜45 分を目安とし、初期セット全体で 6〜9 時間を見込む
- 仕様書 3 件、企画書 3 件、議事録 3 件、CSV 系 3 件を優先する
- アノテーションの差分が大きい文書のみ第三者レビューを入れる

### 運用開始後（半自動蓄積）

- フロントでのノード編集・削除・マッピング承認/却下を `feedback_events` として記録する
- `feedback_events` をそのまま gold に反映せず、週次で `dev` ロールがレビューして採用分のみ `eval/` に取り込む
- 1 件の操作だけでは gold を更新せず、同種の修正が 3 回以上観測されたものを優先的に反映する
- フィードバックが蓄積されるほど評価データが充実する

### `feedback_events` の保存方針

- 生の `feedback_events` の正本は BigQuery に保存する
- repo にはレビュー済みで採用対象になった要約済みイベント、またはそこから生成した正式 eval データのみを保存する
- `feedback_events/*.ndjson` はエクスポート用の補助形式であり、システムの正本ではない
- BigQuery 側では 90 日の詳細保持、月次ロールアップは 1 年保持を基本とする

### gold 更新レビュー方針

- gold データの更新は GitHub PR ベースで行う
- `eval/` 配下の変更は `dev` ロール 2 名のレビューを必須とする
- 差分レビューでは `expected_nodes.json`, `expected_edges.json`, `notes.md` を同時に確認する
- 専用管理画面は初期スコープ外とし、レビュー UI は GitHub の diff を利用する
- 採用された修正だけを repo にマージし、生イベントは BigQuery に残して監査可能にする

### LLM-as-judge（HTML サマリ等）

人手評価が困難な出力（サマリ品質）は Gemini に評価させる。

```
プロンプト例:
  以下のサマリは出典チャンクの内容と整合していますか？
  矛盾点があれば指摘してください。
  [出典チャンク]
  [サマリ HTML]
```

- 評価結果とスコアを `summary_ratings` として保存する
- 評価プロンプトは `source_accuracy`, `coverage`, `structure_quality` を各 1〜5 で返す固定 JSON にする
- 月次で 20 件を人手再評価し、LLM-as-judge の判定精度を監査する

## フィードバックループ

```
プロンプト変更 / コンテキスト設計変更
         ↓
評価データセットで自動評価を実行
         ↓
指標が改善しているか確認
         ↓
改善していれば採用、劣化していれば差し戻し
```

プロンプトや抽出戦略を変更するたびにこのループを回すことで、品質の退行を防ぐ。

### 実行タイミング

- 週次: `eval/` 全体を使った自動評価を定期実行し、結果を BigQuery に保存する
- 変更時: プロンプト、JSON schema、抽出戦略、canonical 化閾値を変更した PR ではマージ前に手動評価を必須とする
- 月次: `summary_ratings` のサンプリング監査と gold データ見直しを行う

### 採用基準

- Precision または Recall が直近 4 週平均より 3pt 以上悪化した変更は差し戻す
- level accuracy のいずれかが 5pt 以上悪化した変更は差し戻す
- ノード統合の誤統合率が悪化した変更は差し戻す
- サマリ評価は `source_accuracy` 平均が 4.0 未満なら要調査とする

## 保存場所

評価データはコードリポジトリに含め、バージョン管理する。

```
eval/
├─ gold_documents/
│  └─ doc_001/ ...
├─ synonym_pairs.json
├─ topic_mapping_ground_truth.json
├─ feedback_events/
│  └─ 2026-03.ndjson
└─ summary_ratings/
   └─ ratings_2026-03.json
```

`feedback_events` は運用ログ、`gold_documents` などはレビュー済みの正式評価データとして分けて扱う。
